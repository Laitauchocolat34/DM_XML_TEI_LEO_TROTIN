1. Édition "critique" numérique des variantes de l'exemplum de la moniale chaste mais bavarde

1.1. Présentation du projet

Ce projet est une proposition d'encodage numérique de l'exemplum de la moniale chaste mais bavarde à l'aide de la Text Encoding Initiative. L'apparat critique est composé de cinq témoins encodés en double-end-point. Grâce à l'encodage déjà réalisé et à la présente ODD qui documente le projet et la méthode à suivre pour la collation, il est possible de compléter l'encodage par l'ajout d'autres variantes de l'exemplum.

L'objectif n'est pas de réaliser une édition critique à proprement parler. Nous n'allons pas comparer les différents témoins manuscrits d'une même oeuvre. Il s'agit plutot de comparer les différentes versions d'un même texte repris par des auteurs.

Ce travail d'XML TEI se situe dans la continuité de notre mémoire de recherche de master. Nous y avons étudié la présence du cadavre dans les récits exemplaires entre le XIIIe et le XIVe siècle.

1.1.1. Présentation du texte encodé

Un exemplum (au pluriel exempla) est une forme de récit bref qui vise à donner un modèle de comportement ou de morale. Il s’agit à la fois d’une fonction rhétorique (codifiée entre autres par Quintilien) et d’un type particulier de récit qui vise à persuader l’auditoire ou le lecteur. Son contenu est très varié, de même que ses emplois : dans les sermons, les ouvrages moraux ou théologiques, les discours juridique. Ses deux principales formes sont l’exemplum rhétorique, porteur d’une morale et d’un modèle de comportement que l’on pourrait qualifier de civique, et l’exemplum homilétique, qui véhicule une morale religieuse et est principalement employé par les prédicateurs à partir du XIIIe siècle.

Ce récit est interessant à étudier sur le long terme puisqu'il a été repris par de nombreux compilateurs d'exempla entre le XIIIe et le XIVe siècle (une vingtaine de versions différentes).

1.1.2. Les témoins choisis

L'édition critique est réalisée à partir de cinq témoins écrits en latin. Nous aurions pu en prendre plus mais les consignes de l'exercice en prévoyait maximum cinq.

Pour réaliser la comparaison nous avons pris la version originelle du texte produite par Grégoire le Grand. Nous l'avons ensuite comparée avec des textes postérieurs provenant de recueils d'exempla du XIIIe siècle. Ce sont surtout les auteurs de recueils d'exempla de cette période qui ont réutilisé ce récit.

Nous avons fait le choix de prendre les récits par ordre chronologique. De commencer par la première mention du récit de la moniale chaste mais bavarde dans les recueils d'exempla du XIIIe siècle puis nous avons suivis l'ordre chronologique.

2. Présentation de l'encodage en XML-TEI de l'exemplum médiévale de la moniale chatse mais bavarde

2.1. La structure du fichier

La présente édition respecte le schéma TEI all.

2.1.1. Structure du teiHeader

Le <teiHeader> est composé de trois sous-ensembles :

  • fileDesc (file description) contains a full bibliographic description of an electronic file.
  • encodingDesc (encoding description) documents the relationship between an electronic text and the source or sources from which it was derived.
  • profileDesc (text-profile description) provides a detailed description of non-bibliographic aspects of a text, specifically the languages and sublanguages used, the situation in which it was produced, the participants and their setting.

2.1.1.1. Structure du fileDesc

Le <fileDesc> est lui-même composé de trois sous-ensembles :

  • titleStmt (title statement) groups information about the title of a work and those responsible for its content.
  • publicationStmt (publication statement) groups information concerning the publication or distribution of an electronic or other text.
  • sourceDesc (source description) describes the source(s) from which an electronic text was derived or generated, typically a bibliographic description in the case of a digitized text, or a phrase such as "born digital" for a text which has no previous existence.

Le <sourceDesc> contient les éléments suivants :

  • listWit (Liste des témoins) contient la liste des témoins utilisés dans cette édition.

L'élément <listWit> contient la liste des témoins utilisées pour faire la comparaison.

La déclaration d'un identifiant, grâce à l'attribut xml:id de l'élément <witness>, permet de se référer à ces témoins dans l'apparat critique. Pour chaque versions du récit nous avons indiqué dans l'élément <bibl> l'auteur, le titre de l'oeuvre, sa datation. Dans un sous élément <bibl> nous avons indiqué l'édition utilisée pour récupérer le texte. Nous y avons indiqué le titre de l'édition, le nom et le prénom de l'éditeur, le lieu de publication, la maison d'édition, la date de l'édition, la page ou se trouve le texte dans l'édition et enfin un lien vers le sudoc.
<witness xml:id="Dialogus_miraculorum">  <bibl>   <author>Césaire de Heisterbach</author>,  <title>Dialogus miraculorum</title>   <date>1219-1223</date>   <bibl>    <title>Dialogus miraculorum</title>, éd.   <editor role="scientific_editor">     <name>      <forename>Joseph</forename>      <surname>Strange</surname>     </name>    </editor>    <pubPlace>Coloniae</pubPlace>,   <publisher>J. M. Heberle</publisher>,   <date>1851</date>,   <biblScope unit="page">p. 193</biblScope>.   <ref target="https://www.sudoc.fr/085299421">Lien vers le Sudoc</ref>   </bibl>  </bibl> </witness>

2.1.1.2. Structure du profileDesc

Le <profileDesc> est composé de deux sous-ensembles :

  • settingDesc (setting description) describes the setting or settings within which a language interaction takes place, or other places otherwise referred to in a text, edition, or metadata.
  • particDesc (participation description) describes the identifiable speakers, voices, or other participants in any kind of text or other persons named or otherwise referred to in a text, edition, or metadata.

Le <settingDesc> contient un index des noms de lieux, renseigné dans l'élément <listPlace>. Il contient aussi des informations sur différents élements du texte. Pour le devoir de XSLT nous avons à faire un index et n'ayant que peu de personnages ou de noms de lieux nous avons du opter pour un index thématique. Nous avons fait le choix de l'élément <setting> qui permet de décrire un contexte linguistique particulier. Ensuite nous avons placé une balise <ab> parce que c'est un bloc anonyme. Enfin j'ai utilisé l'élément <term> pour bien désigner l'élement ou les élements ayant un sens spécifique.

Un identifiant, attribué avec l'attribut xml:id de l'élément <person>, permet d'effectuer des références à l'entrée de l'index dans le corps du texte.

Enfin, une brève description du lieu est renseignée grâce à l'élément <note>.
<place xml:id="monastere">  <placeName>monasterii</placeName>  <note>Un monastère est un ensemble de bâtiments où vit une communauté religieuse de moines ou de moniales.</note> </place>
<ab xml:id="nuit">  <term>Nocte</term>  <note>La nuit, on craint le diable, les brigands, les revenants, les sorcières et les loups-garous.    Rixes, homicides, conspirations, adultères et viols ne sont pas rares</note> </ab>

Le <particDesc> contient un index des noms de personnages, renseigné dans l'élément <listPerson>. Chacun des personnages est nomé comme dans le texte.

Un identifiant, attribué avec l'attribut xml:id de l'élément <person>, permet d'effectuer des références à l'entrée de l'index dans le corps du texte.

Enfin, une brève description de la personne est renseignée grâce à l'élément <note>.
<person xml:id="Gregoire">  <persName>Grégoire le Grand</persName>  <note>Grégoire Ier, était un pape de l'Église catholique romaine qui a exercé son pontificat du 3 septembre 590 jusqu'à sa mort le 12 mars 604.    Il est souvent considéré comme l'un des plus grands papes de l'histoire de l'Église et est vénéré comme un saint dans le christianisme.    Grégoire le Grand a produit une variété d'œuvres écrites qui ont eu une influence significative sur la théologie chrétienne, la pastorale et la liturgie.  </note> </person>

2.1.1.3. Contenu de l'encodingDesc

L'<encodingDesc> donne des précisions sur la manière dont a été encodé le fichier XML qui contient notre édition. Ici, la méthode utilisée est celle du double-end-point. La méthode d'attachement à double point final relie une entrée d'appareil à un texte de base, en l'ancrant aux positions exactes de début et de fin de son lemme dans le texte de base. Elle a été choisit car elle nous paraissait être la plus claire visuelement à utiliser et la plus simple pour nous à maîtriser.
<variantEncoding method="double-end-point"  location="internal"/>

2.2. Méthodologie d'encodage

2.2.1. Choix de la leçon principale

Comme cela a été dit au dessus, le témoin principal est le récit de Grégoire le Grand. C'est le témoin le plus ancien et auquel les auteurs se réfèrent le plus souvent. C'est donc ce témoin qui est encodé dans le <lem>, tandis que les autres témoins sont encodés dans des éléments <rdg>. Lorsque le contenu d'un des autres témoins est similaire au texte de Grégoire le Grand aucun balisage n'est fait. Quand le texte d'un des témoins ne possède pas l'un des passages de la version de Grégoire le Grand, une balise rdg auto-fermente est mise
<app from="#statut_moral2_start"  to="#statut_moral2_end">  <lem wit="#Dialogus">quae <term ref="#sexualite">carnis quidem continentiam</term> habuit</lem>  <rdg wit="#Sermones"/>  <rdg wit="Fabulae"/>  <rdg wit="#Dialogus_miraculorum"/> </app>

2.2.2. Structure du texte et de l'appareil critique

2.2.2.1. Structure du corps de texte

L'exemplum de la moniale chaste mais bavarde est un petit texte de quelques lignes. Nous avons donc fait le choix de ne pas le structurer avec des éléments afin de garder l'intégrité du texte. La structuration du texte c'est principalement faite avec l'encodage en observant les différentes parties du récit. Ces différentes parties ont été nommées dans les xml:id des éléments <anchor>
<anchor xml:id="source_start"/> Vir namque uitae uenerabilis <persName ref="#Felix">Felix Portuensis</persName>episcopus in Sabinensi prouincia ortus atque enutritus est <anchor xml:id="source_end"/>
Pour plus de lisibilité, le texte principal est mis dans les éléments <text>, <body>, <p>. L'apparat critique est quant à lui placé dans un élément <back>.
<body>  <p> <!-- contenu de l'encodage -->  </p> </body>
<back>  <div type="apparatus">   <p>    <app/>   </p>  </div> </back>

2.2.2.2. Choix d'encodage

Comme expliqué précédemment, il ne s'agit pas d'une édition critique à proprement parler. Nous avons surtout orienté notre travail vers la comparaison sémantique et structurelle des récits comme ils sont très différents les uns des autres. Il aurait pu être possible de plus se concentrer sur le choix des mots mais nous n'avons pas assez de connaissances en latin.

2.2.2.3. Imbrication des <app> et modularité de l'édition critique

La présente édition critique suit un principe de modularité : elle permet de voir "les-variations-dans-les-variations". Pour permettre l'étude de variance la plus précise possible, les sous-variations au sein des témoins sont signalés dans un <app> à l'intérieur du <rdg>. L'édition est donc construite en variations/sous-variations.

L'exemple ci-dessous montre une imbrication où un <app> contient
<app from="#gardien5_start"  to="#gardien5_end">  <lem wit="#Dialogus">et pars una</lem>  <rdg wit="#Sermones">qui a renibus supra</rdg>  <rdg wit="#Fabulae"/>  <rdg>   <app from="gardien6_start"    to="gardien6_end">    <rdg wit="#Dialogus_miraculorum">Unde exterriti et videre volentes quid hoc portenderet terram eiecimus et ecce</rdg>   </app>  </rdg> </app>

2.2.3. Caractérisation de l'encodage : utilisation des attributs wit et des éléments <persName> et <placeName> et <term>

2.2.3.1. Identifier les témoins avec l'attribut wit

Placé sur les éléments <rdg> et <lem>, l'attribut wit permet d'identifier les témoins au sein de l'apparat critique.

  • #Dialogus : l'identifiant des Dialogues
  • #Collectio : l'identifiant de la Collectio exemplorum cisterciensis
  • #Sermones : l'identifiant des Sermones Vulgares
  • #Fabulae : l'identifiant des Fabulae et Parabolae
  • #Dialogus_miraculorum : l'identifiant du Dialogus_miraculorum

Ces quatre valeurs font référence à l'attribut xml:id des témoins, qui est donné dans les éléments <witness> du <teiHeader>

2.2.3.2. Identifier les personnages dans le corps du texte avec l'élément <persName>

Dans le corps du texte, les différents personnages mentionnés sont encodés dans un <persName>. Dans le corps du texte (<body>), le <persName> prend un attribut ref dont la valeur est un identifiant. Ces valeurs sont dérivées de l'attribut xml:id
<app from="#source_start" to="#source_end">  <lem wit="#Dialogus">Vir namque uitae uenerabilis <persName ref="#Felix">Felix Portuensis</persName> episcopus in Sabinensi prouincia ortus atque enutritus est</lem>  <rdg wit="#Collectio">De <term ref="#Dialogues">Dialogo</term>   <persName ref="#Gregoire">Gregorii</persName>  </rdg>  <rdg wit="#Sermones"/>  <rdg wit="#Fabulae">narrat <persName ref="Gregoire">Gregorius</persName>  </rdg>  <rdg wit="Dialogus_miraculorum">Dicam vobis rem terribilem quae meis temporibus in villa mea contigit</rdg> </app>

2.2.3.3. Identifier les lieux dans le corps du texte avec l'élément <placeName>

Dans le corps du texte, les différents lieux mentionnés sont encodés dans un <placeName> ; comme avec <persName>, cet élément porte un attribut ref dont la valeur permet de renvoyer au descriptif du lieu situé dans dans le <teiHeader>. Les valeurs de ref sont dérivées du xml:id de chaque lieu, qui est situé dans l'élément <place> du <teiHeader>
<app from="#gardien2_start"  to="#gardien2_end">  <lem wit="#Dialogus">eiusdem <placeName ref="#eglise">ecclesiae custos</placeName>  </lem>  <rdg wit="#Sermones">custos <placeName ref="#monastere">monasterii</placeName>  </rdg>  <rdg wit="#Dialogus_miraculorum"/> </app>

2.2.3.4. Identifier des éléments dans le corps du texte avec l'élément <term>

Dans le corps du texte, les différents éléments à contextualisés sont encodés dans un <term> ; comme avec <persName>, cet élément porte un attribut ref dont la valeur permet de renvoyer au descriptif du lieu situé dans dans le <teiHeader>. Les valeurs de ref sont dérivées du xml:id de chaque lieu, qui est situé dans l'élément <ab> du <teiHeader>
<app from="#gardien_start"  to="#gardien_end">  <lem wit="#Dialogus">   <term ref="#nuit">Nocte</term> autem eadem</lem>  <rdg wit="#Sermones"/>  <rdg wit="Fabulae">In <term ref="#nuit">nocte</term>sequenti</rdg> </app>

2.2.4. Restrictions et contraintes: l'usage de règles Schematron

Une règles Schematron a été crée.

  • Règle contraignant l'usage d'un attribut et ses valeurs : un <rdg> doit contenir un wit dont la valeur ne peut pas contenir '#Dialogus', puisque c'est le témoin principal.

2.3. Les avantages d'un apparat critique nativement numérique

Constituer un apparat critique nativuement numérique de l'exemplum delamoniale chaste mais bavarde présente plusieurs avantages par rapport à un apparat critique classique :

  • Enrichissement et mise à jour de l'apparat. La TEI est un format de conservation pour des données textuelles ; cependant, c'est un format qui peut être enrichi. Il est possible de rajouter d'autres témoins à la présente édition de ce chapitre ; grâce à cette ODD, il est aussi possible d'encoder d'autres exempla/ Enfin, le présent encodage peut être amélioré en lui apportant plus de précision scientifique : la liste de variations possibles pour un <app>, par exemple, peut être affinée ; il est aussi possible d'indiquer la raison de la variation, grâce à l'élément <cause>.
  • Possibilités d'exploitation de l'apparat.
    • Publication. Si la TEI est un format de conservation pérenne des documents, elle peut également servir de format pivot à partir duquel l'apparat encodé peut être transformé et prendre d'autres formes : à l'aide d'une feuille XSL, il est par exemple envisageable de transformer cette édition en une publication sur internet (transformation de XML-TEI vers HTML ou XHTML), d'en faire une publication papier (transformation vers un format dérivé de .TeX) ou encore de transformer l'apparat en un autre document XML (transformation de XML vers XML).
    • Analyses statistiques. Grâce au haut degré de précision et de caractérisation permis par un apparat critique en TEI, il est envisageable de mener une étude des textes encodés. Il serait par exemple possible d'étudier le degré de proximité entre des témoins. Il est aussi possible d'étudier les types de variations, puisqu'elles ont étés caractérisées selon un vocabulaire contrôlé : on peut par exemple mesurer la proportion de chaque type de variation, ou voir quelle variation est la plus répandue. Le but final étant de contextualiser pour chacunes des périodes, des zones géographiques et chacun des auteurs afin de comprendre ces variantes et les expliquer.

<author n="1">  <name ref="http://nzetc.victoria.ac.nz/tm/scholarly/name-427308.html"   type="organisation">New Zealand Parliament, Legislative Council</name> </author>   <author n="2">  <name ref="nzvn:427308"   type="organisation">New Zealand Parliament, Legislative Council</name> </author>   <author n="3">  <name ref="./named_entities.xml#o427308"   type="organisation">New Zealand Parliament, Legislative Council</name> </author>   <author n="4">  <name key="name-427308"   type="organisation">New Zealand Parliament, Legislative Council</name> </author>

In contrast values of the ref attribute are resolved using the widely accepted protocols for a URI, and thus less documentation, if any, is likely required by the recipient in data interchange.

These guidelines provide no semantic basis or suggested precedence when both key and ref are provided. For this reason simultaneous use of both is not recommended unless documentation explaining the use is provided, probably in an ODD customizaiton, for interchange.

Note

The values for this attribute are language ‘tags’ as defined in BCP 47. Currently BCP 47 comprises RFC 5646 and RFC 4647; over time, other IETF documents may succeed these as the best current practice.

A ‘language tag’, per BCP 47, is assembled from a sequence of components or subtags separated by the hyphen character (-, U+002D). The tag is made of the following subtags, in the following order. Every subtag except the first is optional. If present, each occurs only once, except the fourth and fifth components (variant and extension), which are repeatable.

language
The IANA-registered code for the language. This is almost always the same as the ISO 639 2-letter language code if there is one. The list of available registered language subtags can be found at https://www.iana.org/assignments/language-subtag-registry. It is recommended that this code be written in lower case.
script
The ISO 15924 code for the script. These codes consist of 4 letters, and it is recommended they be written with an initial capital, the other three letters in lower case. The canonical list of codes is maintained by the Unicode Consortium, and is available at https://unicode.org/iso15924/iso15924-codes.html. The IETF recommends this code be omitted unless it is necessary to make a distinction you need.
region
Either an ISO 3166 country code or a UN M.49 region code that is registered with IANA (not all such codes are registered, e.g. UN codes for economic groupings or codes for countries for which there is already an ISO 3166 2-letter code are not registered). The former consist of 2 letters, and it is recommended they be written in upper case; the list of codes can be searched or browsed at https://www.iso.org/obp/ui/#search/code/. The latter consist of 3 digits; the list of codes can be found at http://unstats.un.org/unsd/methods/m49/m49.htm.
variant
An IANA-registered variation. These codes ‘are used to indicate additional, well-recognized variations that define a language or its dialects that are not covered by other available subtags’.
extension
An extension has the format of a single letter followed by a hyphen followed by additional subtags. These exist to allow for future extension to BCP 47, but as of this writing no such extensions are in use.
private use
An extension that uses the initial subtag of the single letter x (i.e., starts with x-) has no meaning except as negotiated among the parties involved. These should be used with great care, since they interfere with the interoperability that use of RFC 4646 is intended to promote. In order for a document that makes use of these subtags to be TEI-conformant, a corresponding <language> element must be present in the TEI header.

There are two exceptions to the above format. First, there are language tags in the IANA registry that do not match the above syntax, but are present because they have been ‘grandfathered’ from previous specifications.

Second, an entire language tag can consist of only a private use subtag. These tags start with x-, and do not need to follow any further rules established by the IETF and endorsed by these Guidelines. Like all language tags that make use of private use subtags, the language in question must be documented in a corresponding <language> element in the TEI header.
